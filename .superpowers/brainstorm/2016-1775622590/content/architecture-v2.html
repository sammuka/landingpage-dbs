03 — Estratégia de Endpoints · Compatibilidade + Modernidade

Antes — SOAP único e acoplado

dbs-ws (CXF 2.7.17)
Único endpoint · /dbs · 7 @WebMethod
SOAP/WS-Security com senha hardcoded
Tudo no mesmo processo JBoss
Sem REST · Sem OAuth · Sem retry
BillingServiceBean
JNDI lookup direto
4.700+ linhas de lógica
~300 stubs SOAP para Braspag
BMT manual · sem circuit breaker
Braspag SOAP Outbound
PagadorTransaction · PagadorQuery
CartãoProtegido · ReconciliationFiles
~300 classes geradas por wsimport
Endpoints *.pagador.com.br descontinuados
VS

Depois — SOAP + REST · containers isolados

SOAP Adapter (CoreWCF) — Container próprio
Preserva WSDL original — parceiros sem alteração
WS-Security: senha no Key Vault (não no código)
Delega 100% para Application Layer compartilhada
Deployável e escalável independentemente
REST API (ASP.NET Core) — Novos parceiros
Bearer JWT RS256 via Okta · HTTPS obrigatório
OpenAPI / Swagger documentado
Rate limiting · CORS allowlist · HSTS
Mesma Application Layer do SOAP Adapter
Braspag REST Outbound
POST /v2/sales · PUT /void · PUT /refund
~20 DTOs (records) · substitui ~300 stubs
Polly v8: retry exponencial + circuit breaker
MerchantKey no Key Vault · nunca logado
Plano de Sunset do SOAP Adapter: parceiros notificados no início da migração com 12+ meses de antecedência para migrar para REST. O SOAP Adapter permanece em operação como container isolado até migração completa de todos os parceiros. A decisão de sunset é controlada por feature flag na configuração da aplicação — sem redeploy.
